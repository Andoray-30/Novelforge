# NovelForge 后端整体架构与数据流转指南 (Architecture Map)

本指南是从全局视角俯视整个 **NovelForge** 后端项目源码的“地图”。它将所有散落在各个文件夹内的独立逻辑缝合在一起，展示数据是如何从“用户请求”最终变成“精美的世界书百科”的。

---

## 🏗️ 1. 核心层级概览 (Layer Architecture)

如果我们把项目看作一座工厂，从上到下可以分为 6 个主要层级（以下展示了数据流转和模块间的依赖关系）：

```text
                           [ 用户请求 / 前端调用 ]
                                     |
    +=================================================================+
    |                          1. API 网关层                             |
    |                            /api                                 |
    |                     (接收请求、参数校验、分发)                       |
    +================================+================================+
                                     | (1. 下发任务)
                                     v
    +=================================================================+
    |                       2. 业务调度与主控中心                          |
    |                          /services                              |
    |          (处理大模型调度 ai_scheduler、合并消重 deduplicator)          |
    +========+=======================+======================+=========+
             | (2. 生肉文本切片)      | (3. 分解提取命令)     | (4. 结果组装)
             v                       v                      v
+------------------------+ +-----------------------+ +-----------------------+
|    3A. 文本预处理车间      | |      3B. AI 脑力车间    | |     4. 终端维系组装厂     |
|        /content        | | /extractors & /quality| |      /world_tree      |
| (洗稿、切章、分析物理特征)| |  (各种角色的独立提词器)  | |  (建立实体图谱和超链接)  |
+------------+-----------+ +-----------+-----------+ +-----------+-----------+
             |                       |                       |
             +-----------------------+-----------------------+
                                     | (5. 将生成的 JSON 落库或缓存)
                                     v
    +=================================================================+
    |                         5. 底层仓储系统                            |
    |                           /storage                              |
    |               (统一读写控制网关，写入硬盘或关系型数据库)                |
    +=================================================================+
                                     ^ 
    ===================================================================
                                支撑整个体系
    ===================================================================
    |                       0. 核心法则底座 (Core)                       |
    |                            /core                                |
    |           (所有的模块都依赖这里定义的 Pydantic 数据模型与全家桶配置)       |
    +=================================================================+
```

## 🧭 2. 文件夹逻辑连线 (Directory Reference)

当你在开发时，如果遇到 Bug，请对照下表来精准定位并跳转到对应的模块：

### 🔵 0. `/core` (世界法则与图纸)
- **地位**：全系统依赖度最高、最不能随便改的地方。
- **职责**：里面存放着 `config.py` (API 秘钥和全局速率限制) 和极其重要的 `models.py` (使用 Pydantic 规定了角色、地点、事件表到底长什么样)。**如果你想给角色系统增加一个“战斗力”属性，第一步必须来这里改模板！**
- 🔗 **详情请见**: [core/README.md](./novelforge/core/README.md)

### 🔴 1. `/api` (前台接待窗口)
- **地位**：对外的唯一暴露面（基于 FastAPI）。
- **职责**：处理前端发来的 HTTP 请求，校验参数，然后把任务甩给后面的 `services`。
- 🔗 **详情请见**: [api/README.md](./novelforge/api/README.md)

### 🟣 2. `/services` (主控大脑与包工头)
- **地位**：承上启下，业务最密集的地方。
- **职责**：
  - `ai_service.py` 封装了 OpenAI SDK，真正与大模型通讯的唯一途径。
  - `ai_scheduler.py` 处理几十万字的小说时，大模型会排队。它维护了一个优先级任务池。
  - `*_deduplicator.py` 负责把第三章提取出来的“张三”和第五章的“张三”进行脑补与融合。
- 🔗 **详情请见**: [services/README.md](./novelforge/services/README.md)

### 🟡 3A. `/content` (生肉清洗车间)
- **地位**：只处理物理层面的文本。
- **职责**：小说刚传进来时，通过这里的 `chapter_detector.py` 切割章节，通过 `text_preprocessor.py` 剔除乱码和无用空格，为大模型减轻负担。
- 🔗 **详情请见**: [content/README.md](./novelforge/content/README.md)

### 🟡 3B. `/extractors` & `/quality` (特征提取与质检)
- **地位**：与 AI 深度博弈的 Prompt 工程模块。
- **职责**：`extractors` 中包含了针对各式对象（人物、世界观、关系网）的独立提取器，里面藏着大量的提示词技巧。而 `/quality` 则像是一个质检员，用于给大模型的输出打分，决定要不要重试。

### 🟢 4. `/world_tree` (宏大百科组装线)
- **地位**：小说知识提纯的最后一步。
- **职责**：前面 `extractors` 提取的可能只是一个个孤立的 JSON。这里的 `builder.py` 负责将它们进行“数据库化 JOIN”，将人物绑定到地点，将事件穿插进时间线，形成一颗立体世界树。然后通过 `st_converter.py` 将它翻译成可以导入 SillyTavern 玩语C的 Lorebook。
- 🔗 **详情请见**: [world_tree/README.md](./novelforge/world_tree/README.md)

### 🟤 5. `/storage` (物流仓库)
- **地位**：底层抽象的存储机制。
- **职责**：目前支持把所有中间结果和文本切片变成文件存进本地，未来也很容易拓展为真实的 MySQL/Redis 数据库连接层。
- 🔗 **详情请见**: [storage/README.md](./novelforge/storage/README.md)

## 🛠️ 3. 黄金排错推导流 (Troubleshooting Flow)

以后如果在后端不同分支开发遇到报错，请按以下逻辑推导：

1. **接口报 422 校验错误 / 输出数据结构不对？**
   👉 检查 `novelforge/core/models.py` 的 Pydantic 模型有没有漏掉或拼错字段！
2. **大模型回复的内容老是不符合小说原意？**
   👉 检查 `novelforge/extractors/` 文件夹下的 Prompt 是否没有给够约束。
3. **一本极长的小说提取到一半彻底卡死/请求超时挂掉？**
   👉 检查 `novelforge/services/ai_scheduler.py` 中的异步队列有没有死锁，以及并发阈值（由 `novelforge/core/config.py` 控制）是否太高被 API 拦截了。
4. **乱码文本提取失败 / 章节切分变成了整整一坨？**
   👉 检查 `novelforge/content/chapter_detector.py` 的正则规则是否有疏漏。
5. **在 SillyTavern 里导入后没反应？**
   👉 检查 `novelforge/world_tree/st_converter.py` 中的格式映射规则是否不匹配当前版本的 ST。
